Blue Planet ONOS Installation and Administration Guide

The Blue Planet Open Network Operating System (BP-ONOS) is an open-source software-defined network (SDN) controller for service provider networks. The BP-ONOS SDN controller is designed from the ground up to meet the high availability, scalability and performance requirements of service providers.

BP-ONOS acts as an extensible, modular, distributed SDN controller to manage your network through GUI, CLI, and northbound APIs. The BP-ONOS SDN controller integrates with the Blue Planet Multi-Domain Service Orchestration (BP MDSO) platform and enables network operators to orchestrate network services in the controller domain using intent-driven APIs provided by BP-ONOS.

The Blue Planet MDSO platform does end-to-end orchestration across multiple domains.

BP-ONOS leverages the dynamic open-source community with many users and diverse application bases to give service providers interoperability, security, and protection against vendor lock-in.

BP-ONOS 16.06 features

This release of BP-ONOS offers:

  • High availability through clustering and distributed state management

  • Scalability through clustering and partitioning of network device control

  • Northbound abstractions for:

    • a global network view

    • network graph

    • application intents

  • Pluggable southbound for support of:

    • OpenFlow protocols

  • REST API for northbound abstractions

  • A complete BP-ONOS CLI built upon the Karaf CLI

  • Modern web-based GUI

  • Support for proactive and reactive flow setup

  • SDN-IP application to support traditional IP networks controlled by distributed routing protocols like BGP

Blue Planet ONOS: a carrier-grade architecture

BP-ONOS manages network resources, and provides both abstractions and APIs to manage, monitor, and program network devices. Abstractions and APIs simplify the creation of network applications and services for operation across a variety of devices.

Carrier-grade elements

All these elements contribute to make BP-ONOS a carrier-grade platform.

  • A symmetric and distributed architecture brings scalability and resilience to failures. The ability of BP-ONOS to run as a cluster of instances brings web-style agility and performance. BP-ONOS provides a logically-centralized, simplified control plane so you can build a symmetric cluster of individual instances, with identical software on each instance. In case of failure or when balancing workloads, an instance can take on the workloads of any other instance.

  • System Tiers. Three system tiers separate and define the BP-ONOS layered architecture:

    • The BP-ONOS central core physically separates data and control functions while providing logically centralized views of the network state and network control functions. This core is protocol-agnostic and built on an OSGi/Apache Karaf framework.

    • A southbound abstraction composed of protocol-aware network elements, protocols, network adapters for connectivity, and protocol-specific APIs. Through this abstraction, the core maintains the state of the network without needing to know about the specifics of each network element.

    • A northbound abstraction:

      • Manages and configures services and applications.

      • Provides a global network view. The global network view gives an application a view of the network.

      • Provides an intent framework so that an application can request a service without having to know the details of how the service is performed or routed. A developer or operator specifies an intent through a policy statement or connectivity requirement. The intent framework then determines which requests to accommodate and delivers the requested service to the application.

  • Modularity and separation. The BP-ONOS core platform is modular and provides well-defined boundaries amongst its various sub-systems. This modular construction, as an assembly of sub-systems, enables separation of responsibilities. Modularity and separation make this system easier to maintain, extensible and customizable, and simple to test.

Carrier-grade advantages

A BP-ONOS distributed core provides a highly available environment where network applications do not experience network-related downtime, and where service providers can add capacity without disruption and update system software with zero downtime.

With a layered architecture, the southbound abstraction manages different devices and different protocols without impact to the core. A layered architecture also eases addition and migration of devices and protocols, and insulates the core from details of the different devices and protocols. A device-agnostic southbound abstraction works with legacy devices and is a key enabler for migration to commodity white-box switches.

Furthermore, the northbound abstraction insulates applications from network details not needed by the application. The northbound abstraction also insulates the core from applications, allowing the core to manage requests from multiple, competing applications. This results in faster application development and network changes without application downtime.

Lastly, BP-ONOS software modules make it easy to develop, debug, maintain, and upgrade ONOS.

Blue Planet technologies

This section provides an overview of the Blue Planet ONOS supporting technologies.

SDN

Software-defined Networking (SDN) is an architecture that separates the network control plane from network forwarding functions. Network control is logically centralized onto an SDN controller. The SDN controller is directly programmable and can abstract the underlying infrastructure to create network applications and services. SDN makes it possible to automate network tasks like configuration and policy management and can also enable the network to respond dynamically to changing application requirements.

You can find out more about SDN at Ciena.com and at Open Networking Foundation.

NFV

Network Function Virtualization (NFV) is a technology that decouples network functionality from capacity. NFV abstracts network functions that used to require special-purpose hardware to operate and now creates those functions as software applications on standardized servers, switches, and storage. With NFV, service providers can use less expensive general-purpose hardware and easily scale network functions.

From Ciena’s blog

You can find out more about NFV at Ciena.com.

CORD

The Central Office Re-architected as a Datacenter (CORD), simplifies infrastructure and transforms the central office from having many types of hardware-based appliances to having virtual devices. CORD uses NFV, SDN, and cloud technologies to bring data center economies and agility to the Central Office.

Installing Blue Planet ONOS

Recommended practices

  • Install Blue Planet ONOS through a Docker container that you can find on the Blue Planet Docker registry. Contact Ciena Blue Planet customer support for a login ID and password to access the private registry.

  • The Blue Planet ONOS GUI uses a web browser. Ciena recommends Google Chrome or Mozilla Firefox. Internet Explorer is not supported.

  • This guide assumes that you are installing BP-ONOS on a host server and you will access that server over a network from a remote machine like a laptop.

  • Make sure that you can access the host server with a network protocol like SSH or with a virtual development environment like Vagrant.

  • It is helpful to have a basic understanding of:

    • Installing and using a remote desktop for Linux

    • Installing a basic Linux operating system, like Ubuntu 14.04.04, for example

    • Docker containers

    • SDN

    • OpenFlow, if you need to understand device messaging

    • Mininet, if you are using BP-ONOS for testing and network simulation

Server installation requirements

Blue Planet recommends the following virtual machine (VM) hardware to run BP-ONOS on a host server:

  • 8 CPU cores

  • 32GB RAM

  • 64GB disk space storage

Software prerequisites

Install the following on the host server before you install the BP-ONOS Docker container:

  • Ubuntu Server 14.04.4 LTS - Download Ubuntu.

  • Optional - A GUI desktop compatible with Ubuntu Server 14.04 LTS, like Ubuntu Desktop 14.04.4, for example. If you plan to access a Web browser on the host server or if you are using a PC or Mac in a test environment, then install a Ubuntu-compatible GUI.

An intent is in one of these states:

  • SUBMITTED - The intent is submitted and will be processed soon.

  • COMPILING - The intent is being compiled. This is a transient state.

  • INSTALLING - The intent is being installed.

  • INSTALLED - The intent is installed.

  • RECOMPILING - The intent is being recompiled after a failure.

  • WITHDRAWING - The intent is being withdrawn.

  • WITHDRAWN - The intent is removed.

  • FAILED - The intent is in a failed state because it cannot be satisfied.

How to create a cluster in BP-ONOS

To enable high availability and fault tolerance, run multiple instances of a BP-ONOS controller in cluster mode. All instances in a cluster share state with each other.

This procedure assumes that you are using a Docker container to run BP-ONOS and that you have already downloaded the BP-ONOS image. For details about downloading BP-ONOS through Docker, refer to Download and install Blue Planet ONOS.

To form a cluster, you need:

  • A minimum of three servers, one for each instance

  • An odd number of instances

  • The onos-form-cluster script, which you can download

  • The IP address of each cluster, obtained with the docker inspect command

High-level steps to create a cluster

  1. Optional: Download the onos-form-cluster utility.

  2. Start the BP-ONOs Docker containers.

  3. Get the IP address for each instance.

  4. Form a BP-ONOS cluster.

  5. Log in to BP-ONOS and verify the cluster.

  6. Verify OpenFlow and Proxyarp apps on each BP-ONOS instance.

  7. Optional: Set the network controller to a BP-ONOS instance.

  8. Assign network switches to all instances in the BP-ONOS cluster.

How to create a host-to-host intent

After you’ve connected your network to BP-ONOS, you can create a host-to-host intent.

The intent framework allows an application to request a service without having to know the details of how the service is performed.

This example uses a Mininet network to show how to create a host-to-host intent.

For a successful a host-to-host intent, you must have also completed these actions:

  • Activated the openflow app

  • Activated the proxy arp app

  • Have ONOS discover the network

Note
Both the OpenFlow and proxy ARP apps must be active to create flows on BP-ONOS. By default, both apps are activated at BP-ONOS installation and startup. Refer to the How to activate an app on how to verify active apps. The openflow app allows BP-ONOS to listen for, respond to, and send OpenFlow messages. Appendix A: Set users to secure Blue Planet ONOS

Secure access is available for the BP-ONOS Docker container.

Create a protected user directory

To enable security in BP-ONOS, and to have that security persist across Docker container instantiations, create a directory on the host system to store user information.

Protect this directory so that only root can access it, like /onos-users, for example.

Mount the protected user directory into the Docker container

After you create the user directory, start BP-ONOS and mount this local volume into the Docker container as /users. Mount this volume by using the docker run command with these options, -v /onos-users:/users.

Change the default user

Initially by default, the only BP-ONOS user is karaf with the password, karaf.

To change the default user, run the docker exec command with these options, docker exec <running-onos-container-id-or-name> /root/onos/bin/onos-secure-ssh -u <new-user-name> -p <new-user-password>.

After you set a new default user, karaf/karaf is no longer valid and BP-ONOS is more secure

Add additional users

Note
This release of BP-ONOS does not have multi-level user permissions so any user you add will have all permissions.

To add additional SSH users, use the public SSH key for that user. Enter the Docker command, docker exec aaa /root/onos/bin/onos-user-key <user> "$(cat <users-public-key-file> | awk '{print $2}’)”. You can typically find the public key file a user’s HOME directory under .ssh/id_rsa.pub.

CLI only

After executing the docker exec command, the user can access the BP-ONOS CLI through the SSH command but will not have Web UI access.

Web UI access

For a user to access the BP-ONOS web UI, you must add the user name and password to a file that you can find in the local directory, titled something like, /onos-users/users.properties. Edit this file AFTER you start the BP-ONOS Docker container.
